title: GeoJSON Specification
---

Our GeoJSON specification is based on our CIM profile with some caveats described below in [exceptions](#exceptions).

Most CIM classes can be represented as a GeoJSON `Feature`, with the class, attributes and associations defined in the `properties` object of the Feature.
All `properties` objects must at minimum have a `class` property which should correspond to the desired CIM class. It can then have any CIM field possible
for that particular class according to our [CIM profile](https://zepben.github.io/evolve/docs/cim/evolve/).

All CIM fields are effectively nullable, with many receiving defaults if they are not specified. Omission of a CIM attribute for a type implies that field
is either null or should receive the configured default. You must refer to the SDK for this information.

For all primitive fields the translation between Json types and Java/Kotlin is the following:

|    Json     | Java/Kotlin              |
| ----------- | -------------------------|
| string      | String                   |
| int         | Int                      |
| float       | Double                   |
| boolean     | Boolean                  |
| object      | Map<String, JsonElement\> |
| array       | List<JsonElement\>        |

:::note
At this point we only support objects keyed by strings, and a value can be any valid Json type.
:::

There is some special handling for enumerations, associations, and geometry listed below.
## Enumerations

Any Enumeration property should be specified using its string representation, for example:
```
phases: "ABC"
wireMaterialKind: "aluminum"
vectorGroup: "DYN11"
```

## Associations

Associations between objects are represented as string IDs or lists of IDs and should refer to features defined elsewhere in the GeoJSON,
which should be of the indicated type as per their [CIM documentation](https://zepben.github.io/evolve/docs/cim/evolve/). These fields will use
the name of the association as per the CIM model appended with `Id` or `Ids` as per the [Examples](#examples) sections.

:::warning
There are exceptions to the above rule. Notably Terminals, ConnectivityNodes, and Locations are not present in the GeoJSON spec. See [Exceptions](#exceptions) below.
:::

## Datetimes

Fields with a Datetime/Instant type must be passed as an ISO DateTime string with a UTC offset (`YYYY-MM-ddTHH:mm:ss+00:00`), for example: `2021-04-30T20:30:30`

## Geometry

We currently support Point, LineString, and Polygon geometry types, where Polygon is only used to define an area for a `Substation`. For consistency all
coordinates are represented as an array of arrays, with a Point being an array with a single array inside it containing a double longitude and latitude.
As per the GeoJSON spec, a LineString and a Polygon must have more than one set of coordinates and thus must always be an array containing more than one
array of doubles. Additionally for a Polygon, the first set of coordinates must be _exactly_ the same as the last set.

:::warning
Coordinate sets must be in `longitude, latitude` order.
:::

Some types may also have a null or omitted geometry. It is valid for the following types:

1. Anything extending `AssetInfo`
1. Anything extending `OrganisationRole`
1. Anything extending `Document`
1. Anything extending `Asset`
1. Anything extending `Measurement` or `MeasurementValue`
1. Anything extending `RemotePoint`
1. Anything extending `PerLengthLineParameter`
1. Anything extending `TransformerEnd`
1. `BaseVoltage`
1. `RatioTapChanger`
1. `TransformerStarImpedance`
1. `Organisation`
1. `OperationalRestriction`
1. `Diagram`

## Base GeoJSON structure

The below spec describes the basic structure, where the `properties` object contains all the relevant attributes of the corresponding CIM class,
and the `geometry` providing the longitude and latitude for anything that is a PowerSystemResource.

```json

{
    "type": "FeatureCollection",
    "features": [
        {
            "type": "Feature",
            "id": "string",
            "geometry": null | {
                "type": "string: Point | LineString | Polygon",
                "coordinates": [
                    [
                        0.0,  // longitude
                        0.0   // latitude
                    ],

                    // ...
                ]
            },

            "properties": {
                "class": "string",  // Must be a valid CIM class name (case sensitive)

                // other CIM properties for `class`. See CIM documentation for relevant fields
            }
        },

        // other features
    ]
}
```

## Exceptions

Not all CIM properties are straight 1 to 1 mappings, see the sections below for additional and remapped properties.

### Equipment

Any type that extends `Equipment` has two extra optional properties `circuitId` and `substationId`. These properties can be set on any equipment
to associate it with the corresponding `Circuit` or `Substation`. For example:

```json
{
    "type": "Feature",
    "id": "acls001",
    "properties": {
        "class": "AcLineSegment",
        "circuitId": "CRC001"
    }
}
{
    "type": "Feature",
    "id": "busbar001",
    "properties": {
        "class": "AcLineSegment",
        "substationId": "SUB001"
    }
}
```

### Terminals, phasing, and connectivity

Terminals and ConnectivityNodes are not present in the specification, and instead created from special properties on AcLineSegments.
An AcLineSegment has two properties, `fromId` and `toId` which indicate the connectivity between `ConductingEquipment`. A `fromId` or `toId` is a
string id that points to any other feature defined in the GeoJSON that is a ConductingEquipment, or a PowerTransformerEnd.

Terminals and ConnectivityNodes will be created between each AcLineSegment and the equipment referenced by its `fromId` and `toId`.

Because of this, phases are also recorded on `AcLineSegments` and applied to the generated terminals. A phase on an AcLineSegment can
be any valid [PhaseCode](https://zepben.github.io/evolve/docs/cim/evolve/IEC61970/Base/Core/PhaseCode).

An example of the above exceptions:

```json
{
    "type": "Feature",
    "id": "acls001",
    "geometry": {
        "type": "LineString",
        "coordinates": [
            [
                32.432321,  // longitude
                12.434541   // latitude
            ],
            [
                32.432325,
                12.435423
            ]
        ]
    },
    "properties": {
        "class": "AcLineSegment",
        "inService": true,
        "normallyInService": true,
        "length": 12.0,
        "phases": "ABC",     // Terminals on acls001, breaker001, and tfn001-end1 will get phase ABC.
        "assetInfoId": "cableinfo001",
        "fromId": "breaker001",
        "toId": "tfn001-end1",
        "perLengthSequenceImpedanceId": "plsi001"
    }
}
```

#### Feeders, Circuit, AuxiliaryEquipment

These types all have references to Terminals in some form, however as Terminals are not defined in the specification another method must be used
to refer to the Terminal for each type.

For a Feeder the terminal property will instead refer to the corresponding `ConductingEquipment` for the terminal.

##### Feeder
```json
"properties": {
    "class": "Feeder",
    "normalHeadEquipmentId": "BREAKER001",
}
```
At runtime the `normalHeadTerminal` of the `Feeder` will be the most downstream Terminal of `BREAKER001` based on tracing out from the root of the network.

##### Circuit

```json
"properties": {
    "class": "Circuit",
    "endEquipmentIds": ["BREAKER001", "BREAKER002", "BREAKER003"]
}
```

At runtime the `endTerminals` of the `Circuit` will be the terminals of the breakers closest to the transmission network.

##### AuxiliaryEquipment

An AuxiliaryEquipment supports two equipment properties `equipmentId1` and `equipmentId2`:

```json
"properties": {
    "class": "FaultIndicator",
    "equipmentId1": "PT001",
    "equipmentId2": "ACLS001"
}
```
At runtime the terminal chosen will be the first Terminal from `equipmentId1` that connects to `equipmentId2`.


### PowerTransformerEnd

A PowerTransformerEnd references a Terminal like the above types, however we allow PowerTransformerEnd to be connected to AcLineSegments,
and we use this connectivity to determine which Terminal is associated with the end. This means `toId` and `fromId` on an `AcLineSegment` should prefer to
reference the individual PowerTransformerEnd it's connected to, rather than the PowerTransformer.

### Location

Locations are not defined as a class, and instead can only be sent alongside an existing feature.
A Location will always be inferred from the Features geometry if present, however additional Location fields can be specified in the `properties` of a
GeoJSON object for classes that support Locations. These location properties are represented by an object mapped to the name of the corresponding Location
field on the class. A single Feature can support multiple locations with different names. See the table below for examples:

| Class | Associatian name |
| ----- | ---------------- |
| Anything extending PowerSystemResource | location |
| UsagePoint | usagePointLocation |
| Meter | serviceLocation, location |
| Pole | location |
| Streetlight | location |

A `Location`s PositionPoints are inferred from the features geometry, unless they are indicated via the `positionPoints` field inside the corresponding location
property. The optional `"positionPoints"` accepts the same format data as the geometry. A Feature can have a null geometry (no coordinates) yet still receive
PositionPoints off the location property, however the preferred method is via the Geometry and the `positionPoints` field should only be used when there are
multiple different Locations for a given Feature.

[StreetAddress](https://zepben.github.io/evolve/docs/cim/evolve/IEC61968/Common/StreetAddress) and [TownDetail](https://zepben.github.io/evolve/docs/cim/evolve/IEC61968/Common/TownDetail)
fields for a `Location` should be specified directly inside the relevant location property using their field names. Nesting of StreetAddress or TownDetail
objects is not necessary (see examples).

A Location object can also optionally have an ``"id"` field specified, which will be used as the resulting Locations mRID. If an ID is not present, one will
be generated for you.

For example, the below will generate a Location with mRID `ACLS001-loc` associated with ACLS001 with 2 PositionPoints, associated with a StreetAddress with a
postalCode of 1234, and a townDetail with a name of "123 Smith Street" and a stateOrProvince of "ACT".

```json
{
    "type": "Feature",
    "id": "ACLS001",
    "geometry": {
        "type": "LineString",
        "coordinates": [
            [ 149.098897, -35.285061 ],
            [ 149.098897, -35.285061 ]
        ]
    },
    "properties": {
        "class": "AcLineSegment",
        "location": {
            "id": "ACLS001-loc",
            "name": "123 Smith Street",
            "postalCode": 1234,
            "stateOrProvince": "ACT"
        }
    }
}
```

Below is a Meter with both serviceLocation and location set, with different positionPoints for the serviceLocation:

```json
{
    "type": "Feature",
    "id": "MTR001",
    "geometry": {
        "type": "Point",
        "coordinates": [
            [ 149.098866, -35.284061 ],
        ]
    },
    "properties": {
        "class": "Meter",
        "location": {
            "name": "Telstra Tower",
            "postalCode": 2000,
            "stateOrProvince": "ACT"  // The resulting Meter's location field will receive the PositionPoint from the geometry
        },

        "serviceLocation": {
            "name": "Black Mountain",
            "postalCode": 2000,
            "stateOrProvince": "ACT",
            "positionPoints": [ 149.098813, -35.284023 ]  // the Meter's serviceLocation will receive this PositionPoint
        }
    }
}
```

## Examples

### Network Hierarchy

A simple network hierarchy defining a GeographicalRegion, SubGeographicalRegion, a Circuit and a Loop, and Substations and their Feeders.

```json
{
    "type": "FeatureCollection",
    "features": [
        {
            "type": "Feature",
            "id": "FDR001",
            "properties": {
                "class": "Feeder",
                "normalHeadEquipmentId": "BREAKER001",
                "normalEnergizingSubstationId": "SUB001"
            }
        },
        {
            "type": "Feature",
            "id": "SUB001",
            "geometry": {
                "type": "Polygon",
                "coordinates": [
                    [
                        32.432321,
                        12.434541
                    ],
                    [
                        32.432325,
                        12.435423
                    ],
                    [
                        32.432325,
                        12.435428
                    ],
                    [
                        32.432321,
                        12.434541
                    ]
                ]
            },
            "properties": {
                "class": "Substation",
                "subGeographicalRegionId": "SGR001",
                "normalEnergizedFeederIds": ["FDR001"],
                "loopIds": ["LP001"],
                "normalEnergizedLoopIds": ["LP001"],
                "circuitIds": ["CRC001"]
            }
        },
        {
            "type": "Feature",
            "id": "SGR001",
            "properties": {
                "class": "SubGeographicalRegion",
                "geographicalRegionId": "DNSP001",
                "substationIds": ["SUB001"]
            }
        },
        {
            "type": "Feature",
            "id": "DNSP001",
            "properties": {
                "class": "GeographicalRegion",
                "subGeographicalRegionIds": ["SGR001"]
            }
        },
        {
            "type": "Feature",
            "id": "CRC001",
            "properties": {
                "class": "Circuit",
                "loopId": "LP001",
                "endEquipmentIds": [ /* ... */ ],
                "endSubstationIds": ["SUB001", /* ... */ ]
            }
        },
        {
            "type": "Feature",
            "id": "LP001",
            "properties": {
                "class": "Loop",
                "circuitId": "CRC001",
                "substationIds": ["SUB001", /* ... */],
                "normalEnergizingSubstationIds": ["SUB001", /* ... */]
            }
        }
    ]
}
```

### Connectivity

A basic connectivity model showing PowerTransformerEnds connected to AcLineSegments.

:::note
Geometry is omitted for brevity
:::

```json
{
    "type": "FeatureCollection",
    "features": [
        {
            "type": "Feature",
            "id": "BREAKER001",
            "properties": {
                "class": "Breaker",
            }
        },
        {
            "type": "Feature",
            "id": "ACLS001",
            "properties": {
                "class": "AcLineSegment",
                "phases": "ABC",
                "fromId": "BREAKER001",
                "toId": "PT001-e1"
            }
        },
        {
            "type": "Feature",
            "id": "PT001",
            "properties": {
                "class": "PowerTransformer",
                "powerTransformerId": "PT001",
                "vectorGroup": "DYN11",
                "transformerUtilisation": 0.8
            }
        },
        {
            "type": "Feature",
            "id": "PT001-e1",
            "properties": {
                "class": "PowerTransformerEnd",
                "powerTransformerId": "PT001",
                "endNumber": 1
            }
        },
        {
            "type": "Feature",
            "id": "PT001-e2",
            "properties": {
                "class": "PowerTransformerEnd",
                "powerTransformerId": "PT001",
                "endNumber": 2
            }
        },
        {
            "type": "Feature",
            "id": "ACLS002",
            "properties": {
                "class": "AcLineSegment",
                "phases": "BC",
                "fromId": "PT001-e2",
                "toId": "EC001"
            }
        },
        {
            "type": "Feature",
            "id": "EC001",
            "properties": {
                "class": "EnergyConsumer",
                "grounded": true
            }
        }
    ]
}
```
